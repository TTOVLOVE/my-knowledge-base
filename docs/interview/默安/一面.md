# 默安一面笔试

------

## 1. “`”符号在PHP与SQL中的作用

- **PHP 中（）**：

  反引号在 PHP 中**并不是**无特殊含义的字符串，它是**执行系统命令**的操作符，等同于 `shell_exec()`函数。例如：`$output = ls -al;`。

- **SQL 中（MySQL）**：

  用于**标识符转义**。当表名、列名含有系统关键字或特殊字符时使用，防止 SQL 语法报错。

  例如：`SELECT select FROM user`

## 2. Python语句 'int'.class 含义及漏洞关系

- **含义**：`'int'.__class__` 表示字符串 `'int'` 的类对象，即 `<class 'str'>`。
- **漏洞关系**：这是 **SSTI（服务器端模板注入）** 或 **Python 沙箱逃逸/反序列化 RCE** 的核心基础。攻击者通过不断向上寻找基类（`__bases__`）并向下寻找子类（`__subclasses__`），最终调用危险模块（如 `os`）。
  - 典型 Payload：`''.__class__.__bases__[0].__subclasses__()[117].__init__.__globals__['system']('id')`

## 3. Java中的反射作用

- **作用**：允许程序在**运行时**动态获取类的内部信息（成员变量、方法、构造器等），并能直接操作任意对象的内部属性及方法。

  Java

  ```
  Class<?> clazz = Class.forName("java.lang.String");
  Method method = clazz.getMethod("toUpperCase");
  ```

- **用途**：框架设计（如 Spring）、插件机制、依赖注入。在安全领域，常被用于**构造反序列化漏洞的利用链（Gadget）**，以绕过访问限制执行恶意代码。

## 4. Java反序列化原理

- **原理**：Java 对象通过 `ObjectOutputStream` 序列化成字节流，通过 `ObjectInputStream` 反序列化恢复为对象。
- **漏洞成因**：如果反序列化的数据可控，且类实现了 `Serializable` 接口并在重写的魔术方法（如 `readObject()`、`readResolve()`）中包含危险逻辑。当系统执行反序列化时，这些危险逻辑会被自动触发，从而引发 **RCE（远程命令执行）**。

## 5. SSRF 漏洞利用思路

- **内部服务探测**：扫描内网存活 IP 及端口（如探测 `127.0.0.1`、Redis、数据库）。
- **攻击内网应用**：利用未授权访问打内网接口（如 Kubernetes API、Consul）。
- **云环境元数据获取**：请求云服务商的 IMDS（如 `169.254.169.254`）窃取云凭证。
- **协议走私/扩展**：结合 Gopher/Dict 协议攻击内网 Redis、Memcache 等服务（如 `gopher://127.0.0.1:6379/_`）。
- **组合拳**：与 XSS 结合，或者利用 SSRF 进行本地文件读取（`file://`）。

## 6. 利用 XSS 能达到什么效果？如何防御？

- **攻击效果**：
  - 窃取用户敏感信息（Cookie、Session Token）。
  - 会话劫持，越权操作（常与 CSRF 配合）。
  - 页面伪造、钓鱼攻击、网页挂马。
- **防御机制**：
  - **输入输出处理**：严格的输入格式校验，对输出到 HTML/JS/JSON 的内容进行实体转义。
  - **安全策略**：配置 CSP（内容安全策略）。
  - **Cookie 保护**：敏感 Cookie 设置 `HttpOnly` 标记，防止 JS 读取。
  - **框架保护**：使用主流框架（如 Django、Spring、Vue、React）自带的防 XSS 机制。

## 7. 联合注入如何判断列数（除了order by）

- **不断增加 NULL**：使用 `UNION SELECT NULL, NULL...` 逐渐增加 NULL 的数量，直到页面恢复正常或报错消失。
- **使用 GROUP BY**：使用 `GROUP BY n`（n 为数字），逐渐递增 n，直到出现报错，报错前的值即为列数。

## 8. SQL二次注入漏洞原理

- **第一步（存入）**：用户输入的恶意 payload 经过简单的转义或防护后，被当作合法字符串**插入并存储**到数据库中。
- **第二步（触发）**：应用程序从数据库中读取该 payload，**未经验证**便直接拼接到新的 SQL 语句中执行，导致注入发生。

## 9. PHP反序列化漏洞原理

- **原理**：序列化字符串通过 `unserialize()` 函数恢复为 PHP 对象。
- **漏洞成因**：若反序列化过程中的类定义了魔术方法（如 `__wakeup()`、`__destruct()`、`__toString()`），且这些方法内部调用了危险函数（如 `eval`、`system`）或存在文件操作。只要构造特定的序列化字符串，当对象被唤醒或销毁时，就会触发恶意代码执行（RCE）。

## 10. JWT 是什么？Cookie 和 Session 区别？

- **JWT (JSON Web Token)**：一种基于 Token 的身份认证机制。数据保存在**客户端**，服务端无状态（不保存 Token），通过数字签名保证防篡改。
- **Cookie vs Session**：
  - **Cookie**：数据存储在**客户端**浏览器，每次请求自动携带，安全性较低，容量有限。
  - **Session**：数据存储在**服务端**，安全性高。服务端通过下发一个 `SessionID`（通常存储在客户端的 Cookie 中）来关联用户的状态。

## 11. Shiro550 漏洞原理

- **核心机制**：Shiro 的 `rememberMe` 功能会将用户身份信息进行：`序列化` -> `AES 加密` -> `Base64 编码` -> `放入 Cookie`。
- **漏洞成因**：Shiro 1.2.4 及之前版本使用了**硬编码的 AES 密钥**。
- **攻击链**：攻击者利用已知的默认密钥，逆向构造：`恶意利用链（如 CC 链）` -> `序列化` -> `AES 加密` -> `Base64 编码`，替换原本的 Cookie 发送给服务端。服务端解密并反序列化后，触发 RCE。

## 12. sqlmap --os-shell 原理（MySQL/MSSQL）

- **MySQL**：利用 `SELECT ... INTO OUTFILE`（或 `DUMPFILE`）功能，向拥有写权限的 Web 物理路径下写入一个轻量级的 WebShell 脚本当作后门。
- **MSSQL**：利用 SQL Server 内置的存储过程 `xp_cmdshell` 直接执行系统命令，从而获得一个交互式的 Shell。

## 13. sqlmap 参数：--technique 和 --dbms

- `--technique`：指定要使用的注入技术。
  - `B`: 布尔盲注 (Boolean-based blind)
  - `T`: 时间盲注 (Time-based blind)
  - `U`: 联合查询 (Union query-based)
  - `E`: 报错注入 (Error-based)
  - `S`: 堆叠查询 (Stacked queries)
- `--dbms`：显式指定目标数据库类型（如 `MySQL`、`Oracle`、`MSSQL`、`PostgreSQL`），加快扫描速度并减少发包。

## 16. Redis 未授权访问漏洞利用方式

前提：目标 Redis 暴露在公网且未设置密码认证。

1. **SSH 免密登录**：将本机的 SSH 公钥写入目标服务器的 `/root/.ssh/authorized_keys` 中。
2. **计划任务反弹 Shell**：将反弹 Shell 的命令写入 Linux 的定时任务目录（如 `/var/spool/cron/root`）。
3. **写入 WebShell**：如果已知目标 Web 服务的绝对路径，利用 Redis 写入一句话木马。

## 17. 利用SQL注入写shell的前提条件及写入命令

- **前提条件**：

  1. 当前数据库用户具有 `FILE` 权限（如 root）。
  2. `secure_file_priv` 配置允许导出文件（为空或指定了对应目录）。
  3. 已知 Web 站点的绝对物理路径。

- **写入命令**：

  ```sql
  select '<?php @eval($_POST["cmd"]); ?>' into outfile '/var/www/html/shell.php';
  select '<?php @eval($_POST["cmd"]); ?>' into dumpfile '/var/www/html/shell.php';
  ```
  
- **区别**：

  - `outfile`：导出多行时自动追加换行符，可能会破坏二进制文件。
  - `dumpfile`：严格以字节流形式写入，不会附加任何多余字符，适合写 UDF 动态链接库或二进制文件。

## 21. 对链接 demo.do?DATA=dWlkPTE2== 的测试思路

1. **解码分析**：识别为 Base64 编码，解码后为 `uid=16`。
2. **业务逻辑/越权测试**：尝试变种参数（如改为 `uid=17` 再编码），测试是否存在水平/垂直越权。
3. **注入测试**：对解码后的参数加入 Payload（如 `uid=1' or 1=1--`），重新 Base64 编码后发送，测试是否存在 SQL 注入。
4. **进阶测试**：检查后端接收到 Base64 后的处理逻辑，警惕是否存在 Java 反序列化漏洞，或如果返回 XML 则测试 XXE 漏洞。

## 22. swagger.json 文件测试思路

1. **还原文档**：将 `swagger.json` 导入 knife、Swagger-UI 或 Postman，生成可视化的 API 接口文档。
2. **未授权访问**：遍历测试接口，寻找无需 Token/Cookie 即可直接调用的高权限接口。
3. **参数 Fuzzing**：针对文件上传、命令执行、SQL 注入等潜在功能点进行参数注入测试。
4. **业务逻辑漏洞**：利用获取的接口参数结构，测试越权操作、数据遍历等逻辑漏洞。

## 23. 登录框页面测试思路

1. **暴力破解**：抓包后使用 Burp Intruder 或 Hydra 进行弱口令/账号枚举爆破。
2. **注入测试**：用户名/密码框进行万能密码测试（SQL 注入/NoSQL 注入）。
3. **逻辑漏洞**：测试验证码绕过、忘记密码功能缺陷、注册功能缺陷。
4. **信息泄露**：F12 调试 JS 代码，寻找硬编码的密钥、隐藏的未授权 API 接口。
5. **前端防护绕过**：如果前端有加密，分析加密逻辑进行重放攻击或解密。

## 24. 安卓/iOS App测试思路及工具

- **通信抓包**：使用 BurpSuite、Charles、mitmproxy 进行 HTTP/HTTPS 抓包（需处理证书绕过/SSL Pining）。
- **逆向分析**：
  - 安卓：使用 Jadx、apktool 进行 APK 反编译查看源码和硬编码信息。
  - iOS：使用 Hopper 等工具。
- **动态调试与 Hook**：使用 Frida、Xposed 框架 Hook 关键函数（如加密算法、反抓包检测）。
- **组件与安全检测**：使用 MobSF 进行自动化组件暴露检测、权限滥用分析。

## 25. 微信小程序测试思路

1. **抓包分析**：利用 PC 端微信或微信开发者工具，配合代理抓取接口流量。
2. **源码提取**：在模拟器或 Root 手机中找到小程序的 `.wxapkg` 包。
3. **反编译**：使用 `wxappUnpacker` 等工具将包解压，还原出前端代码与 JS 文件。
4. **深入测试**：在 JS 中寻找硬编码的 `AppSecret`、接口鉴权方式（Token/Sign），针对 API 接口进行常规 Web 渗透测试。

## 26. 给出 IP 的完整渗透测试流程

1. **信息收集**：nmap、masscan、shodan、fofa（端口、服务、操作系统）。
2. **指纹识别**：whatweb、wappalyzer（识别 CMS、框架、中间件版本）。
3. **漏洞扫描**：nuclei、xray、goby、AWVS 进行自动化探测。
4. **Web 测试与爆破**：BurpSuite 手工测试逻辑漏洞/注入，Hydra 爆破弱口令。
5. **漏洞利用**：利用 metasploit (msf)、exploit-db 脚本获取初始权限（GetShell）。
6. **提权维权**：使用 linpeas/winpeas 寻找提权点，植入后门。
7. **内网横向**：建立代理隧道（frp/socat），进行内网渗透。
